# AIDE Reboot Terminology

## Purpose

This reference normalizes reboot vocabulary for Q01-created docs. It does not replace existing governance, support, or capability policy.

## Canonical Terms

- AIDE Core: durable semantics, Contract, Harness, Runtime, Compatibility, Control, and SDK.
- AIDE Hosts: IDE or tool-host shells, host-family lanes, adapters, and proof records.
- AIDE Bridges: bounded connection layers between AIDE Core and local proof profiles, host adapters, generated artifacts, or companion surfaces.
- Contract: profiles, schemas, capability declarations, request and response shapes, and architecture constraints.
- Harness: deterministic validation, fixtures, queue checks, evidence capture, and repeatable proof execution.
- Runtime: future execution substrate; deferred by Q01.
- Compatibility: host, platform, version, migration, support, capability, and blocker posture.
- Control: queue policy, review gates, autonomy boundaries, governance links, and execution records.
- SDK: future authored integration surface; deferred by Q01.
- Dominium Bridge: first planned AIDE Bridge target for the reboot.
- XStack: Dominium's strict local governance and proof profile.
- bootstrap-era: historical P00-P15 repository baseline, including implemented proofs and evidence.
- pre-product: repository state before broad release readiness, packaging maturity, and full verification evidence.

## Usage Rule

Use `planned`, `partial`, `deferred`, `candidate`, or `future` when repository evidence does not support an implementation claim.
