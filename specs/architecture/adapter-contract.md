# Adapter Contract

## Purpose

The host-adapter contract defines the stable interface between any host lane and the shared core.

It exists so that AIDE can keep one shared core while allowing many host adapters to differ in packaging, runtime shape, and capability ceiling.

## Canonical contract concepts

- `host identity`: canonical family, product, technology, support mode, and execution mode
- `host context`: host version, process facts, lane metadata, and runtime constraints known to the adapter
- `document context`: current file or document identity, language, path, content reference, and dirty state if known
- `workspace context`: project or workspace identity and related metadata when available
- `selection/editor context`: cursor, range, active editor, and related text-buffer context when available
- `requested feature`: stable feature id plus arguments and option payload
- `settings config`: shared defaults plus host and feature-specific overrides already resolved into a stable shape
- `diagnostics`: machine-readable issues, warnings, unavailable reasons, or evidence generated by shared or host layers
- `capability report`: current lane capability, target capability, feature availability, and unavailable reasons
- `response outputs`: edits, actions, artifacts, follow-up requirements, and diagnostics

## Required request and response flow

1. The adapter identifies the current host family, technology lane, support mode, and execution mode.
2. The adapter gathers only the context it can honestly provide.
3. The adapter normalizes that context into the shared request envelope.
4. The adapter invokes the shared core through the configured execution mode.
5. The shared core returns a stable response envelope.
6. The adapter translates the response into host-specific UI or runtime behavior.
7. The adapter records or surfaces diagnostics when the request cannot be satisfied.

## Adapter responsibilities

- translate host-native objects into shared contract objects
- avoid embedding reusable business logic that belongs in the shared core
- report capability and unavailability honestly
- preserve request and response ids across the boundary
- apply returned edits, actions, or artifacts using host-specific APIs
- keep exact version coverage out of folder names and inside metadata, matrices, or manifests

## Capability-level expectations

### `L0`

- identify host family, technology lane, and host version where available
- produce a minimal host-capability report

### `L1`

- expose at least one stable command or entry point
- submit a feature request with minimal context

### `L2`

- provide document and editor or selection context
- apply or present document-oriented edits and diagnostics

### `L3`

- provide workspace or project context in the shared shape
- support feature decisions that depend on project-aware inputs

### `L4`

- participate in deeper IDE integration where the host contract genuinely allows it
- still use the shared request and response model instead of inventing lane-local feature semantics

## Ceiling honesty

Capability ceilings differ by host family and adapter technology. A lane may never reach `L4`, and that is an architectural fact rather than a defect.

The contract therefore requires adapters to report what is currently implemented, what is only a target, and why a feature is unavailable.

## Version and naming rule

Exact version coverage lives in metadata, inventory, matrices, and manifests. It does not belong in folder names or adapter-lane names.
