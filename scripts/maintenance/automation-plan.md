# Maintenance Automation Boundary

## Purpose

This document defines what repository maintenance remains manual now and what later phases may automate carefully.

## Manual Now

- roadmap and phase-priority judgment
- interpretation of blocked, degraded, and deferred host-lane posture
- candidate-family promotion decisions
- final cross-document coherence review
- assessment of whether a native proof is honest or only structural

## Good Future Candidates For Automation

- file and directory existence checks
- simple path-allowlist audits
- machine-readable field-shape validation
- basic id cross-reference checks between inventory, manifests, and matrices
- stale placeholder or bootstrap-language detection
- presence checks for run records after implementation waves

## Likely Hybrid Tasks

Some tasks are good candidates for partial automation with human review:

- matrix drift detection
- manifest-to-matrix consistency checks
- report freshness reminders
- environment or lab record completeness checks

## What Should Probably Stay Human-Reviewed

- support-tier and capability interpretation
- roadmap wording and phase boundary decisions
- blocker and deferral classification
- decisions about whether a candidate family is ready for committed host work

## Why Automation Must Stay Careful

AIDE spans modern and historical hosts with uneven evidence quality. Over-automation would make it easier to hide uncertainty, flatten blocked states, or create false confidence. The maintenance goal is higher coherence, not automated optimism.
